# 11 课程大作业设计

## 一、毕业设计的目标

学到这里，前面的知识已经不再只是零散概念。

读者已经接触过：

- Prompt 与 Prompt Engineering
- Tools 与 Skills
- MCP 与外部能力接入
- Agent 的基本控制循环
- 移动端智能体
- Gateway、记忆与检索
- Dify 等低代码工具

因此，课程最后的大作业不再以“补一个函数”或“完成一个小示例”为目标，而是要求围绕一个真实问题，设计并实现一个**相对完整、相对复杂的智能体系统**。

这一章的重点，不是规定所有人必须做成同一种系统，也不是要求必须使用某一种框架，而是希望每个小组能够：

1. 选定一个明确的问题场景；
2. 设计一个真正有任务闭环的 Agent；
3. 根据问题需要选择合适的技术路线；
4. 最终完成可演示、可说明、可复现的项目成果。

> 大作业考察的核心，不是“是否堆了很多技术名词”，而是“是否围绕问题完成了合理的智能体设计”。 

---

## 二、总体要求

课程大作业建议以小组形式完成，每组围绕一个具体场景完成一套智能体系统原型。

这个系统不要求达到工业级完整度，但至少应满足以下几点：

1. 能解决一个真实且具体的问题；
2. 有比较清楚的输入、处理流程和输出结果；
3. 体现“模型决策 + 外部能力 + 控制流程”的结合；
4. 能够进行完整演示；
5. 能够说明自己的设计思路、技术选型和局限性。

换句话说，这次大作业不只是“做一个能聊天的页面”，而是要完成一个**能完成任务的 Agent 项目**。

---

## 三、选题原则

大作业不限定唯一题目，但建议遵循下面三个原则。

### 1. 问题要具体

题目不能停留在“做一个通用 AI 助手”这种过于宽泛的描述上，而应落到一个清楚的问题场景。

例如：

- 面向学生的课程学习助手
- 面向校园生活的出行与提醒助手
- 面向手机使用场景的个人任务助理
- 面向文档处理的资料整理助手
- 面向知识库问答的专业学习助手

一个好的题目，应该能说清楚：

- 目标用户是谁；
- 要解决什么问题；
- 为什么普通聊天机器人不够，需要 Agent 来做。

### 2. 系统要有复杂度

这里说的“复杂度”，不是要求界面复杂，而是要求项目至少体现出一定的任务结构。

例如：

- 不是只做单轮问答，而是要有多步处理；
- 不是只输出一段文本，而是要能调用能力；
- 不是只靠一个 Prompt，而是要有流程组织；
- 不是只演示静态回答，而是要有真实输入与真实结果。

### 3. 技术路线要服务问题

本次大作业允许不同小组采用不同实现路线。

可以基于：

- 自己写的 Agent 框架
- `minimal_agents`
- LangGraph
- Dify
- MCP Server
- 本地 Tool
- 远程 API

这些都可以使用，但是否使用，并不是先验规定，而应由你的问题场景来决定。

> 先明确要解决什么问题，再决定是否需要 LangGraph、Dify、MCP 或移动端能力，这样的技术路线才是合理的。

## 四、推荐方向

为了帮助大家更快进入状态，这里给出几类适合作为课程大作业的方向。

### 1. 移动端个人智能助手

这是最推荐的一类方向，也最适合和本教程前面的内容衔接。

系统可以围绕手机端展开，例如：

- 管理待办事项
- 阅读课程资料并生成摘要
- 结合天气、课程安排和日程生成每日建议
- 通过移动端界面完成自然语言任务输入与结果展示

这类题目的优势在于：

- 场景具体；
- 演示效果直观；
- 容易体现“端上交互 + 后端智能体”的结构；
- 能自然结合 Tool、MCP、记忆和 LangGraph。

> 图片占位：`images/graduation-mobile-case.png`  
> Caption：移动端个人智能助手示意图

### 2. 知识型学习助手

这类项目围绕知识组织、资料检索和内容生成展开，例如：

- 某门课程的问答助手
- 某一领域的资料检索与总结助手
- 能把文档内容整理成学习卡片的智能体

这类题目很适合结合：

- RAG
- Dify 知识库
- 自定义 Tool
- Skill 化的方法模板

### 3. 任务执行型智能体

这类项目更强调流程组织和任务闭环，例如：

- 自动整理文件与生成摘要
- 收集信息并输出结构化报告
- 针对一类任务完成“规划 -> 执行 -> 汇总”

这类项目很适合结合：

- LangGraph
- 自定义控制循环
- MCP 工具接入
- 任务状态管理

### 4. 多入口或多角色系统

如果有小组希望挑战更高复杂度，也可以考虑：

- 一个智能体同时支持网页、命令行、移动端等多种入口；
- 一个系统里包含多个协作角色；
- 不同子任务由不同模块分工完成。

这类方向不是必选项，但如果设计得好，会很有区分度。

## 五、推荐技术路线

这部分不是强制要求，而是给出几种常见路线，方便大家根据题目选择。

### 路线一：基于 `minimal_agents` 继续扩展

适合已经熟悉前面教学代码、希望延续课程示例风格的小组。

可以在此基础上继续加入：

- 自定义 Prompt
- Tool 注册
- Skill 设计
- MCP 接入
- 简单记忆模块
- 移动端前端或命令行前端

优点是与教程衔接自然，结构容易掌控。

### 路线二：使用 LangGraph 组织复杂流程

适合任务明显是多步的、存在条件分支或状态恢复需求的小组。

例如：

- 先规划，再检索，再执行，再汇总
- 调用多个工具并根据结果继续决策
- 任务中需要中断、恢复或保存状态

在这种情况下，LangGraph 往往比单纯手写循环更适合。

但也要注意：

- LangGraph 不是必须使用；
- 如果项目本身很简单，强行上 LangGraph 反而可能增加复杂度。

> LangGraph 更适合“复杂流程需要更清楚地组织起来”时使用，而不是为了使用框架而使用框架。

### 路线三：使用 Dify 快速完成工作流或知识库应用

适合更关注应用设计、知识库问答、流程编排，而不是底层框架实现的小组。

例如：

- 用 Dify 做一个知识库问答助手
- 用 Dify 做一个格式化输出工作流
- 用 Dify 做一个检索增强的课程资料助手

如果系统主体使用 Dify，也完全可以；但需要讲清楚：

- 为什么选择 Dify；
- 工作流是怎样组织的；
- 哪些能力是平台提供的；
- 你的设计创新点在哪里。

### 路线四：混合路线

这也是非常推荐的一类方式。

例如：

- 移动端负责交互；
- 后端 Agent 负责决策；
- MCP 负责接入外部能力；
- LangGraph 负责复杂流程；
- Dify 负责某一类知识库或子任务工作流。

只要结构合理，这样的混合路线完全可行。

## 六、是否必须使用某些技术

这里单独说明几个很容易被误解的问题。

### 1. 是否必须使用 Gateway

不是。

如果项目本身只有一个入口，例如只有一个移动端 App 或一个网页入口，那么完全可以不引入 Gateway。

只有当系统存在：

- 多入口接入；
- 消息需要统一归一化；
- 多个 Agent 或多个后端模块需要统一调度

这类需求时，Gateway 才会比较自然地出现。

### 2. 是否必须使用 LangGraph

不是。

LangGraph 很适合复杂 Agent，但它不是大作业的硬性要求。  
如果你的项目结构并不复杂，自己写清楚控制循环，同样是合理方案。

### 3. 是否必须使用 Dify

也不是。

Dify 是很强的原型工具，但不是所有项目都必须依赖它。  
如果你的项目更偏代码实现，完全可以不用 Dify。

### 4. 是否必须使用 MCP

不是硬性要求，但强烈建议至少体现一种“外部能力接入”的思路。

这个能力可以是：

- MCP
- 自定义 Tool
- 外部 API
- 本地能力封装

关键不是形式，而是要体现：Agent 不是只生成文字，而是能连接真实能力。

> 大作业考察的是系统设计能力，而不是是否勾选了某一份技术清单。

## 七、建议包含的核心模块

虽然不强制规定技术路线，但一个比较完整的 Agent 项目，通常建议至少包含下面几部分。

### 1. 任务入口

系统需要有一个明确入口，例如：

- 移动端页面
- 网页前端
- 命令行界面
- 聊天界面

### 2. Agent 主流程

系统需要有一个清楚的主控制逻辑，例如：

- 接收任务
- 组织上下文
- 调用模型
- 决定是否使用工具
- 汇总结果并返回

### 3. 外部能力

至少应体现一种外部能力接入方式，例如：

- Tool
- MCP
- API
- 检索系统

### 4. 状态或记忆

建议体现一定的状态管理能力，例如：

- 多轮上下文
- 用户偏好
- 历史记录
- 检索式记忆

### 5. 可演示的结果

项目最终必须能进行展示，而不是只停留在设计图。

## 八、毕业设计报告建议包含的内容

最终报告建议至少包含以下部分。

### 1. 项目背景

说明：

- 你想解决什么问题；
- 目标用户是谁；
- 为什么值得做这个系统。

### 2. 需求分析

说明：

- 系统需要完成哪些任务；
- 任务之间是什么关系；
- 哪些能力是必须的，哪些是扩展的。

### 3. 系统设计

说明：

- 整体架构；
- 核心模块；
- 数据流；
- 状态流；
- 外部能力接入方式。

> 图片占位：`images/graduation-system-architecture.png`  
> Caption：毕业设计系统架构图

### 4. 技术选型

说明：

- 为什么选用当前方案；
- 为什么使用或不使用 LangGraph；
- 为什么使用或不使用 Dify；
- 为什么采用某一种 Tool / MCP 接入方式。

这一部分很重要，因为它直接体现了你是否真正理解了课程内容。

### 5. 实现过程

说明：

- 核心 Agent 流程如何实现；
- Tool、Skill、MCP 或 Dify 工作流是怎样接进去的；
- 哪些部分最难；
- 实际做了哪些取舍。

### 6. 演示与结果

建议至少展示：

- 系统界面；
- 一次完整任务的执行过程；
- 核心模块的运行结果；
- 项目最终效果。

> 图片占位：`images/graduation-demo-case.png`  
> Caption：一次完整任务演示截图

### 7. 问题与反思

说明：

- 当前系统还有哪些不足；
- 如果继续迭代，会优先改什么；
- 本项目中最重要的经验是什么。

## 九、建议提交内容

建议每组提交：

1. 项目源码；
2. 项目报告；
3. 运行说明；
4. 演示视频或演示截图；
5. 关键模块说明。

如果项目包含工作流平台或移动端应用，还建议补充：

- Dify 工作流导出文件；
- APK 或安装包；
- 测试样例；
- 使用说明。

## 十、评分建议

课程大作业可以从下面几个维度综合评分。

### 1. 问题定义是否清楚

- 题目是否具体；
- 目标用户是否明确；
- 问题是否真实存在。

### 2. 系统设计是否合理

- 模块边界是否清楚；
- Agent 的主流程是否完整；
- 技术选型是否服务于问题本身。

### 3. 技术实现是否扎实

- 是否真正实现了关键功能；
- 是否体现了智能体系统的复杂度；
- 是否把模型、工具和流程组织起来了。

### 4. 演示是否完整

- 是否能跑通；
- 是否能完成一个完整任务；
- 是否能清楚展示项目价值。

### 5. 报告与表达是否到位

- 文档是否清楚；
- 图示是否清楚；
- 是否能解释设计取舍；
- 是否能说明项目局限与改进方向。

可以参考如下分值分配：

- 问题定义：15 分
- 系统设计：25 分
- 技术实现：30 分
- 演示效果：15 分
- 报告表达：15 分

## 十一、给选题的几点建议

如果一开始不知道做什么题目，可以先问自己下面几个问题：

1. 我想解决的是哪一类真实问题？
2. 这个问题为什么不能只靠一个普通聊天界面解决？
3. 这个系统需要调用哪些真实能力？
4. 这个流程是一轮就结束，还是明显需要多步组织？
5. 这个项目更适合写代码实现，还是更适合用工作流平台实现？

只要把这几个问题想清楚，选题通常就会自然清晰很多。

另外，一个好的题目往往有以下特征：

- 场景清楚；
- 输入输出明确；
- 有真实任务闭环；
- 能体现 Agent 的价值；
- 可以做出可展示结果。

---

## 十二、小结

课程最后的大作业，不是要求每个人都做出同一种系统，也不是要求把所有框架都用一遍。

更重要的是：

> **围绕一个真实问题，设计并实现一个结构清楚、能力明确、能够完成任务的 Agent 项目。**

如果项目中需要复杂流程，可以考虑 LangGraph；  
如果需要快速搭建工作流或知识库应用，可以考虑 Dify；  
如果需要接入真实能力，可以使用 Tool、MCP 或其他方式；  
如果项目偏移动端，也完全可以把手机端作为主要交互入口。

真正重要的，不是使用了哪个框架，而是是否做出了一个值得被称作“智能体系统”的项目。
