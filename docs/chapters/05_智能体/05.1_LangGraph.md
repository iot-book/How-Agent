# 05.1 LangGraph：用图来组织 Agent

## 一、LangGraph 背景

前一章已经介绍了 Agent 的基本组成：

> Agent = LLM + Context + Tools + Control Loop

如果继续往下做，就会遇到一个很实际的问题：

- 当流程只有一轮调用时，自己写一个 `while` 循环已经够用；
- 当流程开始变长，包含条件分支、工具调用、人工确认、状态恢复、记忆保存时，单纯靠手写循环就会越来越难维护。

这正是 LangGraph 的价值所在。

LangGraph 可以理解为一种更工程化的 Agent 组织方式。它不是用来替代大模型，也不是用来替代 Tool 或 MCP，而是用来把这些能力组织成一张“可执行的图”。

用一句话概括：

> LangGraph 解决的问题，是如何把一个复杂 Agent 的执行流程，稳定地组织成节点、状态和边。

---

## 二、LangGraph 是什么

LangGraph 是一个：

> 用来构建、管理和部署长生命周期、有状态 Agent 的底层编排框架。

几个关键词很重要：

- **low-level orchestration**：给你组织流程的底层能力；
- **stateful**：每一步都围绕“状态”来推进，而不是只看当前这一轮输入；
- **long-running**：适合做多步任务、可恢复任务、带人工介入的任务。

可以把 LangGraph 理解成：

-  `MinimalAgent`更像“手写版最小闭环”；
- LangGraph，更像“把闭环升级成图结构后的工程化框架”。

---

## 三、LangGraph 的核心思想

LangGraph 中最重要是下面三个概念：

1. State
2. Node
3. Edge

### 1. State：状态

State 就是整个 Agent 在执行过程中共享的数据。

例如，一个简单系统的状态里可以放：

- 当前用户输入；
- 历史消息；
- 工具返回结果；
- 是否已经完成任务；
- 当前步骤编号。

和前面自己写的 Agent 相比，最大的区别是：

- 在手写 Agent 时，这些信息常常散落在局部变量里；
- 在 LangGraph 里，这些信息会被统一放到 State 中管理。

> State 是 LangGraph 的中心。节点不是各做各的，而是在读写同一份状态。

### 2. Node：节点

Node 可以理解为图中的一个执行步骤。

一个节点可以承担的工作包括：

- 调用一次模型；
- 执行一次工具；
- 检查输入是否合法；
- 根据结果决定是否继续；
- 对状态做一次更新。

因此，节点不是“页面组件”，也不是“模型本身”，而是：

> 一段围绕状态进行处理的可执行逻辑。

### 3. Edge：边

Edge 表示“下一步去哪里”。

最简单的情况是顺序执行：

- 开始 -> 模型节点 -> 工具节点 -> 输出节点

更复杂时，也可以有条件分支：

- 如果模型决定调用工具，就走工具分支；
- 如果模型已经得到答案，就直接结束；
- 如果需要人工确认，就转到人工审核节点。

> Edge 决定流程怎么走，Node 决定每一步做什么，State 决定整个流程围绕什么数据运转。

## 四、最小 LangGraph 例子

官方仓库 README 给出了一个非常小的工作流示例。它很适合用来理解 LangGraph 的最基本写法。

```python
from langgraph.graph import START, StateGraph
from typing_extensions import TypedDict


class State(TypedDict):
    text: str


def node_a(state: State) -> dict:
    return {"text": state["text"] + "a"}


def node_b(state: State) -> dict:
    return {"text": state["text"] + "b"}


graph = StateGraph(State)
graph.add_node("node_a", node_a)
graph.add_node("node_b", node_b)
graph.add_edge(START, "node_a")
graph.add_edge("node_a", "node_b")

app = graph.compile()
print(app.invoke({"text": ""}))
```

输出结果为：

```python
{"text": "ab"}
```

这段代码虽然简单，但已经把 LangGraph 最基本的几件事展现出来了：

1. 先定义 State；
2. 再定义 Node；
3. 然后用 Edge 把 Node 连起来；
4. 最后 `compile()` 成一个真正可执行的图。

> 这一例子没有大模型，没有 Tool，也没有记忆，但已经完整体现了 LangGraph 的基本工作方式。

## 五、为什么 LangGraph 适合 Agent

如果只是做一个“一问一答”的聊天程序，其实没必要专门上 LangGraph。  
但一旦进入真实 Agent 场景，它就会开始变得很合适。

### 1. 因为 Agent 天然就是多步流程

一个真实 Agent 往往不会只执行一步，而是会经历：

- 理解任务；
- 选择是否调用工具；
- 执行工具；
- 把工具结果写回上下文；
- 继续推理；
- 最终返回答案。

把这些步骤画出来，本来就像一张图。

### 2. 因为 Agent 天然依赖状态

Agent 要做的很多事情，本质上都离不开状态：

- 当前消息历史是什么；
- 前一步调用了什么工具；
- 工具返回了什么结果；
- 当前是不是已经得到最终答案；
- 是否要继续下一轮。

LangGraph 的设计本来就是围绕状态推进，所以它和 Agent 的需求非常契合。

### 3. 因为 Agent 经常需要中断、恢复和记忆

在简单示例里，程序通常是一口气跑到底。  
但在真实系统中，可能会出现：

- 模型返回后要人工审核；
- 服务中途失败后要恢复；
- 一次对话结束后要保留历史；
- 多个线程要分开保存状态。

LangGraph 官方把这类能力放在“persistence（持久化）”体系里处理，这也是它比普通手写循环更工程化的地方。

> LangGraph 并不是为了“把循环写得更花哨”，而是为了把复杂 Agent 中真正难维护的部分收拢起来。

## 六、StateGraph 应该怎样理解

在 LangGraph 里，最常用的构造方式就是 `StateGraph`。

官方文档对它的理解可以概括为：

> 节点通过读取和更新共享状态来通信。

这句话很关键，因为它说明 LangGraph 的节点之间并不是直接把结果手工传来传去，而是统一围绕 State 更新。

例如，一个问答 Agent 的状态可以设计成这样：

```python
from typing import Annotated
from operator import add
from typing_extensions import TypedDict


class AgentState(TypedDict):
    user_input: str
    answer: str
    tool_results: list[str]
    messages: Annotated[list[str], add]
```

这里可以看出一个很典型的 LangGraph 设计思路：

- 某些字段是覆盖式更新，比如 `answer`；
- 某些字段是累积式更新，比如 `messages`、`tool_results`。

这比把所有东西硬塞到一个字符串提示词里更清楚，也更适合工程维护。

> StateGraph 的真正价值，在于它让“复杂流程的数据怎么流动”这件事变得可见了。

可以把它们类比成下面这样：

| 视角 | minimal-Agent | LangGraph |
|---|---|---|
| 关注重点 | 基本闭环怎么跑起来 | 多步流程怎么组织 |
| 典型形式 | `while` 循环、工具调用、消息回填 | 状态、节点、边、持久化 |
| 更适合什么 | 教学入门、最小实现 | 复杂工作流、长链路任务 |

> LangGraph 不是推翻前面的 Agent 讲法，而是把前面的控制循环升级成更适合工程化扩展的结构。

## 七、LangGraph 中的预置 Agent

如果自己从 `StateGraph` 开始搭，能更清楚理解底层原理；  
如果想更快做出一个可运行 Agent，LangGraph 也提供了预置组件。

官方 `prebuilt` README 中给出的一个典型例子是 `create_react_agent`：

```python
from langchain_anthropic import ChatAnthropic
from langgraph.prebuilt import create_react_agent


def search(query: str):
    """Call to surf the web."""
    if "sf" in query.lower() or "san francisco" in query.lower():
        return "It's 60 degrees and foggy."
    return "It's 90 degrees and sunny."


tools = [search]
model = ChatAnthropic(model="claude-3-7-sonnet-latest")

app = create_react_agent(model, tools)

app.invoke(
    {"messages": [{"role": "user", "content": "what is the weather in sf"}]}
)
```

这个例子说明两件事：

1. LangGraph 既可以从底层图结构自己搭；
2. 也可以直接使用上层封装好的 ReAct Agent。

## 八、LangGraph 的记忆与持久化

LangGraph 官方文档中非常强调 persistence，也就是持久化能力。

它的核心思想是：

- 图在运行时，可以把每一步状态保存为 checkpoint；
- 这些 checkpoint 按线程 `thread_id` 组织起来；
- 这样系统就能支持恢复、中断、回看历史，以及跨轮次记忆。

官方文档给出的 Python 示例可以简化理解为：

```python
from langgraph.graph import StateGraph, START, END
from langgraph.checkpoint.memory import InMemorySaver
from typing import Annotated
from typing_extensions import TypedDict
from operator import add


class State(TypedDict):
    foo: str
    bar: Annotated[list[str], add]


def node_a(state: State):
    return {"foo": "a", "bar": ["a"]}


def node_b(state: State):
    return {"foo": "b", "bar": ["b"]}


workflow = StateGraph(State)
workflow.add_node("node_a", node_a)
workflow.add_node("node_b", node_b)
workflow.add_edge(START, "node_a")
workflow.add_edge("node_a", "node_b")
workflow.add_edge("node_b", END)

checkpointer = InMemorySaver()
graph = workflow.compile(checkpointer=checkpointer)

config = {"configurable": {"thread_id": "1"}}
graph.invoke({"foo": "", "bar": []}, config)
```

这里最关键的不是具体字段名，而是两个概念：

- `checkpointer`
- `thread_id`

`checkpointer` 负责保存状态；  
`thread_id` 负责区分不同会话或不同任务线程。

这和前面教程里的“记忆与检索”章节刚好能对应起来：

- 前面的章节强调“记忆是什么”；
- LangGraph 这里强调“状态如何在流程里被保存和恢复”。

> 在 LangGraph 里，短期状态并不是附加功能，而是整个执行框架的核心能力之一。

## 九、小结

这一节最重要的不是记住某个 API 名字，而是建立下面这个认识：

> **LangGraph 本质上是在用 State、Node 和 Edge 来组织一个复杂 Agent 的执行过程。**

从教程的上下文来看，可以把它看成：

- 不是一个新的大模型；
- 不是一个新的 Tool 系统；
- 不是 MCP 的替代品；
- 而是一种更适合工程化 Agent 的流程组织框架。

如果后续要继续深入移动端智能体、多工具协同、长任务执行或多轮状态恢复，LangGraph 会是一个非常值得掌握的方向。
